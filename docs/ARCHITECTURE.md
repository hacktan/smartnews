# SmartNews — Architecture & Master Plan

> **SmartNews — News that checks its claims**
>
> Created: 2026-04-04
> Status: ACTIVE — new stack, building from scratch on top of migrated codebase

---

## 1. Core Philosophy

SmartNews is not a generic RSS reader. It is a **high-signal, claim-verified news discovery platform**.

The central differentiator: **cross-source claim verification**. When the same story is covered by multiple sources, SmartNews extracts specific factual claims from each article, compares them, and surfaces contradictions, agreements, and gaps. This is not a layer added on top — it is the foundation the product is built around.

Everything else (hype scoring, credibility ranking, entity navigation, AI briefing) supports this core mission.

## 2. Stack

| Layer | Technology | Replaces | Why |
|-------|-----------|----------|-----|
| Pipeline runner | GitHub Actions (cron) | Azure Databricks Jobs | Free, no cold start, no cluster management |
| Database | DuckDB | Delta Lake + Unity Catalog | Local, zero-cost, same SQL syntax |
| DB persistence | GitHub Releases | ADLS / Delta storage | Pipeline uploads `.duckdb` after each run |
| AI enrichment | OpenAI API (gpt-4o-mini) | Azure OpenAI | Direct, cheaper, same models |
| Embeddings | OpenAI text-embedding-3-small | Azure OpenAI | Same model, direct API |
| API | FastAPI → Render.com | Azure Container Apps | Same code, free tier |
| Frontend | Next.js → Vercel | Azure Container Apps | Same code, free forever |
| Dev workflow | Claude Code + MCP | Manual CLI | GitHub MCP + DuckDB MCP |

**Total cost: ~$1-3/month** (OpenAI API only)

## 4. Data Architecture — Medallion (DuckDB)

Same Bronze → Silver → Gold → Serve pattern, now in DuckDB instead of Delta Lake.

```sql
CREATE SCHEMA bronze;   -- raw ingested data, never modified
CREATE SCHEMA silver;   -- cleaned, categorized
CREATE SCHEMA gold;     -- AI-enriched, scored, claim-extracted
CREATE SCHEMA serve;    -- UI-ready, pre-computed, full overwrite per run
```

### Key Tables

#### Bronze
| Table | Description |
|-------|-------------|
| `bronze.raw_feeds` | Raw RSS entries |
| `bronze.article_fulltext` | Full-text scraped content (trafilatura) |

#### Silver
| Table | Description |
|-------|-------------|
| `silver.articles` | Cleaned articles, categorized, deduped |

#### Gold
| Table | Description |
|-------|-------------|
| `gold.news_articles` | Enriched: scores, summary, entities, embedding |
| `gold.story_groups` | Cross-source story groupings (3-tier matching) |
| `gold.article_claims` | Extracted claims per article (GPT) |

#### Serve (full overwrite per run)
| Table | Description |
|-------|-------------|
| `serve.article_cards` | UI-ready article cards |
| `serve.article_detail` | Full article detail + related |
| `serve.category_feeds` | Per-category feeds |
| `serve.trending_topics` | Topic aggregations |
| `serve.story_clusters` | KMeans clusters |
| `serve.entity_index` | Entity × article mappings |
| `serve.daily_briefing` | GPT 5-bullet daily digest |
| `serve.compiled_stories` | Multi-source compiled stories |
| `serve.claim_verdicts` | Compared claims with verdict ← NEW |

## 5. Core Feature — Claim Verification

This is the primary differentiator. Every architectural decision should support this.

### How It Works

```
Story matching identifies: "3 sources covered the same story"
        ↓
04b_claim_extraction.py
  → GPT reads full text of each article
  → Extracts atomic factual claims:
     "Company X raised $500M" / "CEO said Y" / "Launch date is Z"
        ↓
Claim comparison across sources:
  → Same claim appears in all 3? → CONSISTENT
  → Claim in 2 sources with different numbers? → DISPUTED
  → Claim only in 1 source? → UNVERIFIED
        ↓
serve.claim_verdicts:
  story_id | claim_text | sources[] | verdict | confidence
```

### New API Endpoint
```
GET /api/story/{id}/claims
→ Returns claim list with per-source breakdown and verdict
```

### New Frontend Page
```
/story/[id]
→ Headline + all covering sources side by side
→ Claims highlighted with color-coded verdict badges
→ "3 sources agree" / "1 source disputes the number" / "Only reported by X"
```

### Claim Extraction Prompt (draft)
```
Given this news article, extract all atomic factual claims — 
specific, verifiable statements of fact (numbers, dates, names, events).
Do not include opinions or predictions.
Return as JSON array: [{claim, type, confidence}]
Types: STATISTIC | QUOTE | EVENT | DATE | ATTRIBUTION
```

## 6. Pipeline — GitHub Actions

```yaml
# Runs every 4 hours, or on manual trigger
schedule:
  - cron: '0 */4 * * *'
workflow_dispatch:

steps:
  1. Download smartnews.duckdb from GitHub Releases (or start fresh)
  2. Run 01_bronze_ingestion.py
  3. Run 01b_fulltext_scraping.py
  4. Run 02_silver_transformation.py
  5. Run 03_gold_aggregation.py
  6. Run 03b_story_matching.py
  7. Run 04_ai_enrichment.py         (incremental — enriched_at IS NULL)
  8. Run 04b_claim_extraction.py     (incremental — claims_extracted IS NULL)
  9. Run 05_serving_projection.py    (rebuilds all serve tables)
  10. Upload smartnews.duckdb to GitHub Releases
```

**Cost**: 0 minutes for public repo (unlimited). ~9 min/run × 6 runs/day = ~1,620 min/month for private (under 3,000 min free limit).

## 7. DuckDB vs Spark — Key Differences in Code

```python
# OLD — Databricks/Spark
df = spark.read.table("bronze.raw_feeds")
df.filter(df.language == "en").write.saveAsTable("silver.articles")

# NEW — DuckDB
import duckdb
conn = duckdb.connect("smartnews.duckdb")
conn.execute("""
    INSERT INTO silver.articles
    SELECT * FROM bronze.raw_feeds WHERE language = 'en'
""")
```

```python
# OLD — Spark MLlib for clustering
from pyspark.ml.feature import HashingTF, IDF
from pyspark.ml.clustering import KMeans

# NEW — scikit-learn
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.cluster import KMeans
```

All SQL logic is identical. Only the compute layer changes.

## 8. MCP Development Workflow

With `.mcp.json` configured, Claude Code can:

| Action | Before MCP | After MCP |
|--------|-----------|----------|
| Check pipeline output | Write query, you run it | Claude queries DuckDB directly |
| Trigger pipeline run | You open GitHub Actions | Claude triggers via GitHub MCP |
| Check Actions status | You open browser | Claude reads run logs |
| Create PR | You run gh CLI | Claude creates via GitHub MCP |

```json
// .mcp.json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "<pat>"
      }
    },
    "duckdb": {
      "command": "uv",
      "args": ["run", "mcp-server-duckdb", "--db-path", "./smartnews.duckdb"]
    }
  }
}
```
